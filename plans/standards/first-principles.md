# First Principles

Think from these foundations, not from "what do other apps do?" Every design, code, and business decision should trace back to one of these. These are heuristics, not specifications — one line beats a paragraph, a name beats a definition, constraints beat instructions.

**PageLines Operating Principles** are load-bearing rules unique to this codebase. **Universal principles we apply** are the cross-disciplinary heuristics that have actually shaped decisions here. **Code & Engineering** are operational rules for the daily refactor / review loop. Each is a one-liner with the reason, not a textbook treatment — go to the source if you want the full theory.

---

## Quick Reference

| Aspect | Rule |
|---|---|
| **Default posture** | Delete, compress, and simplify before adding structure. |
| **Source of truth** | Put each fact where it physically lives; derive or observe everywhere else. |
| **Code shape** | Pure transforms, DI'd state, thin controllers. |
| **Tests** | Every production bug gets a regression check at the cheapest layer that catches it. |
| **Observability** | Visibility is part of the feature; a hidden root cause is a bug. |
| **Reports** | One evidence artifact can feed many views; second digests are parallel systems. |
| **Docs** | Capture why, decisions, and gotchas; source owns values and commands. |

## PageLines Operating Principles

These principles are non-negotiable. Every architectural decision and every PR is judged against them. Each links to its detailed treatment.

- **Multi-tenant query scoping.** Every query returning user-visible data is scoped by ownership or membership. Unscoped queries are security vulnerabilities — "public" means visible within your tenant, not across them. See [core.md → Multi-Tenant Query Scoping](core.md#multi-tenant-query-scoping). Why: a single missed scope leaks customer data.
- **Simplicity is our strength.** Less than normal is the default. Title without subtitle. Header without intro paragraph. CTA without tagline. README without positioning paragraph. JSDoc without restating the signature. Confirmation dialog without restating the verb. The typical version is a bias to defend against, not a baseline to meet — *Compression* below is the technique, this is the posture. The first question on any surface, file, or doc is *what can be deleted before this earns its keep?* Why: inherited scaffolding ("everyone has a subtitle, so we need one") rarely survives inspection, and every element a reader has to skip is a tax on the moment we needed their attention.
- **Source of truth is closest to physics — query external state, don't store it.** The layer where a fact physically lives is authoritative; everywhere else is a pointer, a pure derivation, or an observation. Fly knows what machines and apps exist, Stripe knows what's billed, and the agent runtime knows what files exist. Store irreducible references (`customerId`, provider org IDs) and observations (`lastHealthyAt`); derive names from identity (`agentId` + env) and query the source for current state. See [core.md → Query External State](core.md#query-external-state-dont-store-it). Why: stored copies go stale silently; bugs stay invisible until production.
- **Fail fast on writes, degrade gracefully on reads.** Mutations throw with actionable errors; reads return absent and log so the caller decides policy. Silent fire-and-forget catches are forbidden where they hide bugs. See [core.md → Fail Fast Principle](core.md#fail-fast-principle). Why: silent failures cascade into confusing unrelated explosions later.
- **Heuristics over specifications.** When writing rules — for code, for docs, for bot prompts — give the principle and the reason, not the step-by-step script. Over-specification does the thinking the reader should do. Drop to specification only for safety-critical or deterministic plumbing. See [prompting.md → Heuristic Prompting](prompting.md#heuristic-prompting) and [skills.md → First Principles](skills.md#first-principles). Why: specifications break on edge cases; heuristics generalize.
- **No brittle facts in docs.** Never document values you can `grep` or `cat` from source — version numbers, config values, file paths. Docs capture **why**, **gotchas**, and **decisions**. A version pinned in a guide goes stale the day someone bumps the Dockerfile. Why: drifting docs are worse than no docs — they confidently misinform.
- **Automate until manual testing adds net signal.** Every bug catchable by a test must be caught there first. Manual click-through exists only for UX judgment, brand feel, and aesthetic feedback — things a machine genuinely can't evaluate. See [e2e.md → E2E Journey Toolkit](../testing/e2e.md#e2e-journey-toolkit). Why: manual checks don't scale, don't diff, and miss the 3am regression.
- **Design for the non-technical first-time visitor.** Every surface is judged by how a regular person understands it in 30 seconds: *what is this*, *what to do next*, *where's the value* — without learning vocabulary. No jargon as action labels. No copy written to the product about itself. See [e2e.md → Judge rubrics](../testing/e2e.md#e2e-journey-toolkit). Why: the customers we need to acquire are not us; momentum dies in vocabulary-learning.
- **Security serves momentum.** Build real boundaries — tenant scope, owner checks, encryption, no secret logs — then choose the lowest-friction path inside them. If a security design makes normal users think like admins, tighten the architecture instead of adding chores. Why: a "safe" flow users abandon fails both security and UX.
- **Momentum through curated choices, not magic.** One obvious forward action per screen; pre-build options instead of blank fields. Curated choice = ownership without cognitive tax. Never auto-decide — magic erodes the "I did it" moment; let the user trigger, but make the path obvious. See [e2e.md → Judge rubrics](../testing/e2e.md#e2e-journey-toolkit). Why: less friction ≠ fewer clicks — it's removing the moments the user has to pause and think.
- **Always give feedback — no magic, no disappearing states.** Every action produces a visible response; every failure a named one. Spinners that vanish, swallowed errors, silent state changes — all bugs. Errors carry structured codes so messages can be tailored, not generic. Why: ambiguity erodes trust faster than the error itself. See [api.md → ApiResponse Pattern](api.md#apiresponse-pattern).
- **Interruptions earn their right — never double-feedback.** A push, modal, or sound is a tax on attention; skip it when an in-app surface already informed the user. *"Did this just buzz at me for the message I'm reading?"* is a first-impression failure that erodes trust faster than any single bug. Always-give-feedback is one half; don't-double-feedback is the other.
- **No yak-shaving — every change earns its keep.** Push something forward: more revenue, a launch, more stability, less code, faster feedback for the user. If you can't say in one sentence what this change moves, don't start. Why: every line touched is a chance to break something; cosmetic work spends risk budget on no upside.
- **Surface follows job-to-be-done.** The primary surface owns the start path and one-thumb writes (single-field edits, identity changes, one-tap state); the secondary owns multi-step forms, OAuth, payments, and multi-row admin (bulk operations, permission changes, billing). **Test:** *"from the bus, 30s, one thumb?"* — yes → primary; needs typing or a five-field form → secondary, deep-linked. Blocker-removal duplicates on both. Why: over-stuffing the primary defeats on-the-go; stranding the start path forces a context switch into the surface you just opened.

## Universal principles we apply

The handful of cross-disciplinary heuristics that have actually shaped PageLines decisions. Each lives here because it has a recurring trigger in our work — not as background reading.

- **One concept per view; one obvious next action.** Cognitive load kills conversion. Hick's Law is the *why* (decision time grows with options), data-ink ratio is the *how* (every pixel earns its place). Fires every time we draft a screen — see [`design.md`](design.md), [`features/onboarding.md`](../features/onboarding.md).
- **Activation energy beats persuasion.** Make the right thing the easiest thing — Google OAuth before email/password, prefilled options before blank fields, defaults that cover the common case. If a desired action requires explanation, the friction is the bug.
- **Loss aversion frames upgrades.** People work harder to avoid losing than to gain. "Keep your data" beats "get features" — see [`standards/pricing.md`](pricing.md), [`features/commerce.md`](../features/commerce.md) if it exists.
- **MVP, then "what does this look like at the limit?"** Ship the smallest thing that proves the hypothesis. Then walk the chain: if A, then B, then C — what breaks at scale? The chain catches premature scope; the MVP catches premature optimization.
- **Compression — the best UI/code/explanation is the most compressed version that loses nothing essential.** Apply to docs, prompts, error messages, function signatures alike.

## Planning & Implementation

When designing a feature or fix, before any code:

- **Plan in three layers — pure transforms, DI'd state, thin controllers.** Pure transforms (args in, return out, no hidden inputs) in `utils/<concept>.ts`; stateful collaborators on a class with constructor DI; controllers parse → call → shape. If you can't draw this picture, the design isn't ready. Every transform in a pure util pays back in every future test. See [`refactoring.md → The primary direction`](refactoring.md).
- **TDD specifies behavior; the impl is the side effect.** Write the failing test first for new endpoints, behavior changes, bug fixes. Skip only for pure renames. A test written after the impl describes what the impl does, not what it should do. Same rule for prompts ([`bot-first-principles.md → Building & Iterating`](bot-first-principles.md)).
- **Every production bug ships a regression check at the cheapest layer that catches it.** Type → schema → unit → integration → E2E → behavior judge. Fix is incomplete until the check exists. The gap that let the bug through stays open until you close it.
- **Single source of truth — one canonical instance per domain object.** Global stores (Pinia-style, module-level mutables, "shared state" singletons) are globals with extra steps; views reading different copies will disagree. State belongs on a DI'd client (`AgentClient`, `UserClient`, `ServicesClient`) — views consume from there, never instantiate their own. Why: parallel copies of the same state are AI-coded entropy in disguise; the bug surfaces weeks later as a phantom UX issue nobody can reproduce. Globals also defeat testability — you can't inject a fake into a module-import. See [`code-style.md → No shadow state`](code-style.md) and [`code-style.md → Late-bound DI for genuine cycles`](code-style.md).
- **No monkey patching as architecture.** If code only works because a test or plugin rewrites a module/global/runtime object after load, the dependency is hidden. Use DI, pure helpers, or a documented boundary wrapper. Monkey patches are upgrade-hostile and make failures non-local. Exceptions require explicit confirmation for an unmodifiable third-party boundary, and they stay isolated.
- **State explanations never dead-end.** A system/auth/billing/provisioning message names the state in plain English, gives the exact next action, and includes a link when a link exists. Prefer one sentence plus one action over technical diagnosis. If no link exists, say the in-place recovery ("try again", "wait a moment", "ask the owner"). Why: "upstream error, check logs" makes the user feel trapped; a state message should create momentum.
- **Observability is a first-line feature.** Design for diagnosis on day one: every failure mode names itself with a structured code, every async edge logs, every critical path has phase timing. Adding visibility *after* a prod incident is the wrong order. Smell: bumping a timeout without first answering *"what would have told me WHY it was slow?"* Why: invisible failures are bugs of equal weight to the original — file the visibility gap before the symptom fix. This is the operational counterpart to "Always give feedback" — the bot tells the user; the system tells the operator. See [`observability.md`](observability.md).
- **Subtract before adding.** Before a new file, function, branch, or abstraction — ask "what can I delete or extend instead?" Insert-only PRs are the AI failure mode; the human imposes subtraction. The cheapest abstraction is the one we don't write.
- **Search before naming.** Before exporting a new symbol, grep for synonyms (`fetchUser`/`getUserById`, `Agent`/`AgentRecord`, `userId`/`accountId`). Same-role synonyms become parallel systems that drift — the #1 entropy source in AI-coded codebases.
- **No tutorial-shaped abstractions.** No interface with one implementor. No factory for one constructor. No strategy / provider / plugin before the second consumer. Extract on the third real call site, not the second.
- **Summit of experts when stuck.** When investigation has stalled, a design decision is genuinely contested, or a freeze/regression evades single-agent analysis — consult more than one specialized agent in parallel. Different model lineages produce different candidates. Share one brief; compare verdicts; ship the convergent answer or the contested one with cited evidence. See [`expert-summit.md`](expert-summit.md). Why: a single agent can plateau on its own first hypothesis; two well-briefed agents converge faster than one iterates.

> **AI-review blind spots covered above.** AI agents struggle most with non-local effects: globals, fire-and-forget, magic state changes, cross-module interactions. The mitigations live across these principles — *Single source of truth* + DI in `core.md` (no globals), *Honest signatures* in Code & Engineering below (no fire-and-forget), *Always give feedback* + *No magic* in Operating Principles (no silent state changes), *Search before naming* + *One concern, one place* (no acausal duplicates). When reviewing AI-authored code, scan for these specifically — they don't surface in line-by-line review the way logic bugs do.

## Code & Engineering

- **Fail fast, loud, near the source.** Throw at the point of detection with enough context to diagnose without cross-system log-chasing. Silent failures cascade into confusing unrelated explosions; a crash tells you what broke, a swallow tells you nothing.
- **Honest signatures.** A function that does async work returns `Promise<T>` — `void promise` discards rejection and is fail-fast in disguise. Effects belong at the edges; a `void`-ed promise mid-stream is an untracked background task. At sync boundaries with no honest place to send a rejection (event handlers, watchers), make the callback `async` and wrap the body in **one** try/catch. Log the caught value directly — don't pre-unwrap and lose the stack.
- **Detect preventable failures before expensive actions.** Before anything costly — provisioning, external API calls, long-running jobs — run the cheap check that could catch the known failure class. A 200ms reachability probe saves a 3-minute crash-loop. Preflight rules must be shared across all callers (create form, admin action, start command) so they can't drift. When a new outage mode appears, the first fix is "what preflight would have caught this?" — behavioral fix second. See `bot-api/preflight.ts` and `provisioner/preflight.ts`.
- **Visibility is not optional.** Every failure mode needs enough signal to diagnose without shelling into live infra. An invisible root cause is a bug of equal weight to the original failure — file it as "visibility gap: X is silent, add Y" before papering over the symptom. Never bump a timeout without first answering "what would have told me WHY it was slow?"
- **One evidence artifact, many views.** A run, audit, or report produces one canonical evidence stream; summaries, timelines, cost tables, and CI annotations derive from it. A second renderer-owned digest is a parallel system that can disagree with the truth. Why: duplicated aggregation makes debugging the report as hard as debugging the product.
- **A timeout is rarely the actual problem.** When a test, request, or operation times out, the timeout itself is almost always masking a real bug — a swallowed error, a hung promise, a silent retry loop, an upstream that returned 200 but never streamed bytes, a missing response frame, an endpoint that never gets called. **Bumping the timeout is the wrong first move.** It hides the bug for one more cycle, then surfaces somewhere costlier (prod, on a customer). Default playbook before touching a timeout: (1) trace the request end-to-end — does *every* expected hop appear in logs? (2) look for the silent failure — caught-and-discarded errors, fire-and-forget promises, missing structured-error responses. (3) name the upstream's actual failure mode — model 401, SSE never closing, validator rejecting, network blip with no retry. The timeout is the right fix only after you can describe in one sentence WHY it took that long, and the answer is "this work genuinely takes that long" — not "something silently broke and the test waited until it gave up." When in doubt, prefer to surface the failure faster (shorter timeout + structured error) over hiding it with a longer one.
- **Verify before speculating.** Name the theory, state what evidence would falsify it, look for that evidence in the source — yours or upstream — before shipping the fix. 30 min reading code beats 3 hours guessing. When the proposed fix's shape doesn't match the known failure mode of the thing it's blaming, the theory is probably incomplete. Budget: if you've speculated >30 min, stop and read. Note uncertainty in commits and TODOs; don't present speculation as diagnosis.
- **Simpler beats cleaner.** A rename beats a directory split. One table beats two. A flag beats a new type. "Cleaner separation" that doubles the surface area isn't cleaner — it's just more. YAGNI > premature abstraction; DRY, but three copies before you extract; SOLID only when it earns its complexity.
- **Types are the cheapest assertion.** Push detection compile-time before test-time. `as any` on an owned type is a missing field, not a cast. Compiler-caught beats CI-caught beats prod-caught — orders of magnitude in cost. Same idea as preflight: catch the error class as far upstream as it'll go.
- **Reduce entropy on every PR.** *Entropy* = the cumulative complexity tax: parallel systems, fragmented sources of truth (same fact in two places that can drift), naming drift, dead branches, transforms trapped in controllers, override flags, **magic** (non-causal effects — globals, fire-and-forget promises, cross-system automation without a visible trigger), untested seams. Reducing it isn't only LOC down — sometimes LOC goes up (a tested util, a Zod boundary, an engine array, a DI seam) while entropy drops because the surface area is *understood* and *contained*. Mechanics that lower it: single source of truth per fact, DI'd state (no magic), honest signatures (no fire-and-forget), pure transforms with explicit inputs, engine arrays over branching, contextual organization (group by domain, not type), inversion of control, DRY at three call sites, SOLID as restraint, YAGNI on the speculative, tests pinning behavior. A useful question on any PR: *what got smaller — runtime LOC, concept count, coupling, or sources of truth?*
- **Consolidate type-fragile contracts; inline shallow wrappers.** Same "reduce entropy" verb, opposite verdict by case. **Consolidate** when the same fact is stringly-typed across modules — a URL path, an error code, a wire shape — and the type system can't catch drift; one method on one owner is the only place truth lives (`/api/agents/*` → `AgentClient`, OpenAI/Stripe/Fly → owning provider class). **Don't consolidate** when a "helper" would just rename a one-line local expression: `interface FooArgs` for a single function, a `utils/foo-api.ts` that wraps one template literal, a re-export that just shuffles paths. The import + the file + the one-impl interface ARE the entropy — they fragment one fact into three places to read. Decision rubric for any extraction: *would a future reader prefer to scroll, or to follow an import?* If the wrapped expression is short, named after what it returns, and the wrap adds no shared logic (auth, retry, error mapping, parsing) — leave it inlined. Threshold to extract: **≥3 callers** AND shared behavior beyond the value being constructed. The two reified rules: [`code-style.md → Function Arguments Pattern`](code-style.md#function-arguments-pattern) (inline args type, no single-use `interface FooArgs`), [`code-style.md → Endpoint consumption`](code-style.md#endpoint-consumption--one-method-per-endpoint-on-the-owning-client) (one method per endpoint, but scope to ≥3 callers + shared behavior).
- **Docs evolve with the code.** When code changes, the relevant spec updates in the same PR — prune stale plans, sharpen rationale, remove what's no longer true. The Operating Principle *No brittle facts in docs* covers the *don't-document-grep-able-values* rule; this principle covers the *prune-as-you-go* mechanic that keeps the rest of the doc honest.
- **Refactor = subtract, then move.** The first question in any refactor is "what can be deleted?" — before "what should be split?" Moves without deletes just relocate entropy. If a refactor adds more surface than it removes, it should be a smaller refactor. _"The most common mistake a smart person makes is doing something really well that should never have been done at all." — Elon Musk_
- **No backwards-compat by default — opt in when needed.** Replace cleanly, delete the old shape, update callers in the same commit. Opt into staged rollout only when a paying customer, a separate-team consumer that can't change in lockstep, or an explicit ask requires it. Why: "easy to remove later" is a lie — every fallback layer is permanent the moment it ships.
- **One concern, one place.** If a behavior takes three files or three config fields to understand, collapse into one callback. Don't split "what credentials" from "how to authenticate" from "where to send" — one function builds the complete request.
- **Minimize dependencies.** Contextual organization (group by domain, not by type). Every line is a liability, every UI element cognitive load, every feature maintenance. The best code is code you didn't write.
- **Secrets are secrets.** Encrypt at rest, never log, never return in API responses. Plaintext secret = same severity as plaintext password.
- **Env vars are for secrets and per-env values, not constants.** Static values (demo codes, allowlists, fixed toggles) belong in code — version-controlled, type-checked, grep-able. **Test:** *would this value genuinely differ between dev/test/prod?* If no, it's a constant. Why: every env var is a coupling that drifts silently — pay that cost only when the value varies per environment.
